# 强大功能详解 → 从入门到精通

LLM Wiki 不止是简单的笔记存储，它有很多强大的功能帮你构建活的知识库。这篇教程带你深入了解。

---

## 🌿 多知识库 → 知识隔离，互不干扰

### 什么时候需要多个知识库？

| 场景 | 分开更好 | 放一起更好 |
|------|----------|-----------|
| 工作 + 个人 | ✅ 分开 | - |
| 多个不同项目 | ✅ 分开 | - |
| 不同领域（编程 + 历史 + 投资） | ✅ 分开 | - |
| 同一个领域所有笔记 | - | ✅ 放一起 |

### 实际操作示例

**创建工作知识库：**
```
/llm-wiki init ~/Documents/work-knowledge "工作知识库"
```

**创建个人知识库：**
```
/llm-wiki init ~/Documents/personal-notes "个人笔记"
```

**查看所有知识库：**
```
/llm-wiki list
```

输出：
```
📚 已配置的知识库:

* [当前] 工作知识库 → /Users/you/Documents/work-knowledge
  个人笔记 → /Users/you/Documents/personal-notes
```

**切换到个人知识库：**
```
/llm-wiki switch "个人笔记"
```

输出：
```
✓ 已切换到: 个人笔记
```

**查看当前状态：**
```
/llm-wiki status
```

输出：
```
当前知识库: 个人笔记
路径: /Users/you/Documents/personal-notes

统计:
- 概念页面: 23
- 来源页面: 12
- 问答页面: 8
- 总计: 43 个页面
```

## 🎯 项目整体摄入 → 把整个代码库变成架构知识库

这是 LLM Wiki 最强大的功能之一 —— 把整个代码项目摄入进来，自动提炼架构知识。

### 什么时候用？

- 接手了一个新项目，想快速理清架构 → 摄入，然后问问题
- 自己维护的项目，想沉淀设计决策 → 永久保存
- 读开源项目源码，边读边整理知识 → 日积月累

### 实际操作

假设你刚克隆了一个开源项目 `~/code/redis`，想理解它的架构：

```
# 先切到你放项目知识的知识库
/llm-wiki switch "开源项目学习"

# 摄入整个项目
/llm-wiki ingest --project ~/code/redis -y
```

### 它会自动做这些：

1. **跳过不需要的**：`.git`、`node_modules`、`build` 产物、二进制文件都自动跳过
2. **分类处理**：
   - README 和文档 → 完整提取
   - 核心源文件 → 提取模块结构、关键函数、注释说明
   - 测试文件 → 默认跳过
3. **提炼概念**：自动识别核心模块，为每个创建独立概念页面
4. **建立链接**：自动用 `[[概念]]` 链接关联相关页面

### 摄入完怎么用？

直接提问：

```
/llm-wiki query "Redis 的整体架构是分层的吗？每层负责什么？"
```

```
/llm-wiki query "object 文件中主要定义了哪些核心数据结构？"
```

```
/llm-wiki query "持久化 RDB 和 AOF 分别在哪个模块实现，核心流程是什么？"
```

LLM 会基于它提炼好的知识给你整合回答，比你自己翻代码快多了！

### 自定义忽略规则

如果你想忽略更多目录，在知识库根目录创建 `.llm-wiki-ignore`：

```
# .llm-wiki-ignore 示例
vendor/
tests/
*.md~
*.tmp
docs/old/
```

每个规则一行，语法：
- `目录名/` → 匹配整个目录
- `*.ext` → 匹配扩展名

## 🔄 知识复利 → 问答保存，越用越聪明

### 什么是知识复利？

> 你问了一个好问题，得到了一个好答案，把它存进 Wiki。下次再问这个问题，直接就能拿到整理好的答案，不需要重新推导。而且这个答案还能被其他查询引用，知识越来越连贯。

### 使用示例

你有一个关于分布式锁的问题：

```
/llm-wiki query "用 Redis 实现分布式锁有哪几种方案，各自优缺点是什么？"
```

LLM 基于你之前摄入的几篇文章，给了你一个非常清晰的回答，分了表格对比。你觉得这个回答太棒了，以后肯定还会看。

LLM 问你：
> 这个问答有长期保存价值吗？是否需要保存为 `wiki/answers/Redis实现分布式锁对比.md` ？

你回复：
```
是的，保存
```

结果：
- 创建了 `answers/Redis实现分布式锁对比.md`
- 添加到了 `index.md` 索引
- 将来任何人问相关问题都能用到这个整理好的回答

这就是复利 —— 一次整理，反复使用。

## 🏥 Lint 自动维护 → 让知识库保持健康

用了一段时间后，知识库难免会有些"亚健康"问题。`lint` 帮你找出来，还能自动修复。

### 检查哪些问题

| 问题类型 | 说明 | 能自动修复吗 |
|----------|------|-------------|
| **孤立页面** | 没有任何其他页面链接到它 | ❌ 你决定要不要保留 |
| **死链** | 链接到不存在的页面 | ❌ 需要手动删除或创建 |
| **索引缺失** | 页面存在但没加进索引 | ✅ `--fix` 自动添加 |
| **缺失概念** | 多个页面都提到了，但它自己没独立页面 | ✅ `--fix` 创建占位页 |
| **内容矛盾** | 不同页面说法不一样 | ❌ 需要你判断哪个正确 |
| **需要扩展** | 内容太简短，讲得不够清楚 | ❌ 需要你补充 |

### 使用示例

```
/llm-wiki lint
```

输出：
```
## 健康检查结果

### 静态问题
- ⚠️ 孤立页面: answers/旧问题讨论.md
- ✗ 死链: 没有发现

### 语义问题
- ⚠️ 缺失概念: 红锁（被提到 4 次，没有独立页面）
- ⚠️ 需要扩展: 分布式锁 内容只有一句话

### 修复建议
使用 `/llm-wiki lint --fix` 自动修复：
- 创建 红锁 占位页面
- 添加缺失页面到索引
```

你看了报告，然后：

```
/llm-wiki lint --fix
```

输出：
```
✓ 已创建 concepts/红锁.md 占位页面
✓ 已添加红锁到 index.md
✓ 修复完成

请手动完善：
- concepts/红锁.md 目前只有占位内容，有空补充一下
```

### 多久一次？

建议每个月跑一次 `/llm-wiki lint --fix`，知识库就一直健康。

## 🧩 结合 Obsidian → 双重好

LLM Wiki 产出就是纯 Markdown，用 Obsidian 格式的内部链接，天生和 Obsidian 兼容。

### 推荐工作流

```
你的知识库目录 → 同时也是 Obsidian Vault
```

这样：

| 工作 | 用谁 |
|------|------|
| 摄入新项目、提炼知识、查询问答 | Claude Code + LLM Wiki |
| 日常浏览、手动编辑、笔记整理、反向链接查看 | Obsidian |

### 需要什么配置吗？

不需要！直接把知识库目录打开当 vault 就行，Obsidian 原生支持 `[[链接]]` 语法。

### 示例结构

```
~/Documents/my-wiki/          # LLM Wiki 知识库
├── CLAUDE.md
├── raw/
└── wiki/
    ├── index.md
    ├── concepts/
    ├── sources/
    └── answers/
```

在 Obsidian 打开 `~/Documents/my-wiki/wiki/` 或者整个 `my-wiki/` 都可以。

## 💡 高级使用技巧

### 技巧 1: 按日期组织 untracked

你可以在 `raw/untracked/` 下按日期建子目录：

```
raw/untracked/
├── 2026-03/
│   ├── article-1.md
│   └── article-2.md
└── 2026-04/
    └── new-note.md
```

然后 `/llm-wiki ingest --all` 会全部处理，处理完会移到 `raw/ingested/YYYY-MM-DD/`。

### 技巧 2: Ingest 增量更新

如果你更新了原始文件，重新放去 `untracked` 再 ingest 一次，LLM Wiki 会：
- 识别这是更新
- 合并新知识到已有页面
- 更新索引
- 日志记录更新

不用你手动删旧文件。

### 技巧 3: 大项目分批次摄入

如果你有一个很大的项目，几千个文件，不要一次性全 ingest：

```bash
# 先摄入核心模块
/llm-wiki ingest --project path/to/project/src/core -y

# 过两天再摄入业务模块
/llm-wiki ingest --project path/to/project/src/business -y
```

这样 LLM 处理质量更好，不会因为上下文太大而出错。

### 技巧 4: 保留原始源

Ingest 完成后原始文件自动移到 `raw/ingested/` 保留着，不会删除。你什么时候想重新提取，都可以拿出来再处理。

### 技巧 5: Git 版本控制你的知识库

因为都是纯文本文件，非常适合用 Git 管理：

```bash
cd ~/Documents/your-wiki
git init
git add .
git commit -m "Initial commit"
```

然后推去 GitHub private 仓库，多设备同步，历史回溯都有了。

## 🔮 设计理念：为什么要这么做？

### 对比传统笔记

| 传统笔记 | LLM Wiki |
|----------|----------|
| 你负责整理一切，很累 | LLM 帮你做簿记，你只负责方向 |
| 笔记之间链接靠你手动加 | LLM 自动建立交叉链接 |
| 放着不动就过时了 | 不断更新，知识复利 |

### 对比传统 RAG

| RAG 系统 | LLM Wiki |
|----------|----------|
| 每次查询都重新解析原始文档 | 解析一次，结构化后反复使用 |
| 没有"记忆"，每次从零开始 | 优质问答保存，知识越来越丰富 |
| 需要向量数据库，复杂 | 纯文件，不需要额外基础设施 |

**核心就是一句话：** 让 LLM 做它擅长的整理归纳，你专注在思考和提问。
